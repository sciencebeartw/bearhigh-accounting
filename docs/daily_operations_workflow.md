# BearHigh 日常帳務與老師薪資 SOP

最後確認：2026-07-22

這份文件記錄使用者真正的日常工作方式，供後續 Codex 視窗、其他 AI 與維護者接手。任何新的帳務介面、資料模型或自動化，都必須先讀本文件與 `docs/numbers_workflow.md`，不得只依程式現況自行推測。

## 1. 工作目標與真相來源

BearHigh APP 要取代 Numbers 成為日常操作入口，重點不是重做一套會計系統，而是讓以下三件事連成同一條資料流：

1. 學生何時加入或退出哪一門課。
2. 每一堂實際應計薪的人數。
3. 學生收退費與老師薪資。

核對來源的角色如下：

- `bearhigh` APP：完成切換後的學生、選課、應收、收退費、課程名單、堂次與薪資主檔。
- 高中部學生名冊 Numbers：截至 2026-07-22 的正式切換基準，以及歷史帳務／薪資查核來源。
- Google Sheet `115學年度 山熊升大 學生資料`：學生姓名、學校、學年度課程與行政名單的核對來源；若本次任務另有明確指定唯一真相，以該次指定為準。
- Google Calendar `山熊科學高中部`：每班實際上課日期與堂次的第一個核對來源。
- 行政老師點名單：實際上課日期、到班名單、新增與退出的第二個核對來源。

不同來源不一致時，不可偷偷選一個值覆蓋。姓名重複、老師不明、特殊套餐、特殊堂次或日期衝突都要停下來讓使用者確認。

## 2. 每月老師薪資作業

每月出薪資時，逐班照以下順序處理：

1. 選擇結算月份，列出該月所有需要結算的課程與老師。
2. 核對 Google Calendar 的班級事件，確認本月上課日期與堂數。
3. 再核對行政老師點名單上的日期；Calendar 與點名單不一致時標示差異，不自動猜測。
4. 核對每個日期的點名人數是否符合 APP／原 Numbers 名單。
5. 找出本月新增學生、退出學生及其生效日期。
6. APP 依每堂日期與生效日，自動算出當堂有效人數。
7. 依該課程已確認的老師薪資規則，逐堂計算授課所得。
8. 同一老師的所有課程加總，產生單一老師薪資條與全部老師總表。
9. 薪資條下方列出本月異動：誰在何日入班、誰在何日退班、影響哪一班。
10. 覆核特殊堂次、老師／科目對應及總額後，儲存不可被後續名單變動覆寫的月結快照，再列印或存 PDF。

### 堂次與人數的生效規則

- 入班日當堂開始計入人數。例如 6/15 入班，6/15 當堂人數 `+1`。
- 退班日是第一堂不再計入的日期。例如 6/15 退班，6/15 當堂人數 `-1`。
- 課程的 `firstSessionDate` 是整門課第一堂；學生的 `firstClassDate` 是該生開始計入該課人數的日期，兩者不可混用。
- 每一堂都要保留日期、堂次、基準名單、當堂加入、當堂退出、計薪人數與特殊備註，不能只存整月平均人數。
- Calendar 出現「不收費」、補課、停課或其他特殊註記時，原文保留並標示「請覆核」；APP 不自行判斷是否仍支付老師薪資。

## 3. 新生或舊生加科

1. 搜尋現有學生，確認不是同名或重複資料；找不到才建立新學生。
2. 建立或補齊姓名、基本資料、年級／學年度與學校。
3. 選擇學期、課程、科目、老師、課程第一堂日期及學生第一個計入日期。
4. 登記該科原價、合報／套餐、實際分攤金額、付款日期、付款方式及本次實收。
5. 同一次確認要同步建立／更新學生、選課、應收、收款流水與入班事件，不能只完成其中一半。
6. 完成後，學生要自動出現在該老師的課程名單。
7. 下次結算薪資時，APP 自動在生效日顯示「新增某學生」，且從該堂起人數 `+1`。

若同一學生一次報多科，各科都要保留自己的實際收入與課程生效日。合報總價不能只存在備註裡。

## 4. 舊生退班與退費

1. 搜尋學生並選定要退出的課程，先確認同名與學期。
2. 依點名單／堂次日期計算退班前已上的堂數。
3. 設定退班生效日；該日當堂開始不計入老師薪資人數。
4. 退掉科目的已上堂數，原則上每堂收 `1000` 元。
5. 若原本是合報，依退班後剩餘科目數重新定價，不可直接把單科餘額退回。
6. 顯示原已收、剩餘科目重定價、已上堂數費用與預計退款，讓使用者確認。
7. 同一次確認要同步更新退費流水、應收、選課狀態、退班事件、老師課程名單與稽核紀錄。
8. 後續每一堂該班人數自動 `-1`；當月薪資條下方列出退出姓名與日期。

一般新制合報：

| 退班後科目數 | 應收總額 |
|---:|---:|
| 1 科 | 21,600 |
| 2 科 | 42,200 |
| 3 科 | 61,800 |
| 4 科 | 81,400 |

一般舊制合報：`16,800 / 32,600 / 47,400 / 62,200`。

一般退費公式：

```text
退款 = 本學期／套餐已收
     - 退班後剩餘科目重新定價
     - 退掉科目已上堂數 × 1,000
```

高三學測／分科有年度與科目組合差異。若無法從已確認定價表或 golden sample 唯一推回，APP 必須要求人工輸入剩餘學費並再次確認，不可套用一般 21,600 元規則硬算。

## 5. 同科換班與暫時換班

學生可能因學校進度或程度需要，在同一科目的兩個老師／班級之間暫時或永久換班。這不是退班，也不是重新購買一科：補習班仍只收該期 24 堂的原學費，例如新制一科 `21,600` 元。

真實換班不可直接覆蓋原課程，否則當月無法知道哪幾堂應算給原老師、哪幾堂應算給新老師。APP 應建立一筆換班事件，至少包含：

- 學生、科目與學期。
- 原課程／原老師。
- 新課程／新老師。
- 生效日期。
- 暫時或永久。
- 暫時換班的預計返回日期，或日後再建立一筆換回事件。
- 換班原因，例如學校進度。

計算規則：

- 生效日前的原班堂次計入原老師；生效日當天起，原班不再計入該生。
- 生效日當天起的新班堂次計入新老師。
- 暫時換回時，用換回生效日做相反方向的同一組規則。
- 同科換班不新增學費、不退費、不重算合報，也不增加學生已購買的 24 堂總權益。
- 老師薪資依兩個班各自實際上課日期與有效人數計算；薪資條下方列出「某日某學生由 A 班轉入 B 班」。
- 已儲存的歷史月結快照永遠不回算；換班只影響尚未結算或生效日之後的堂次。

只有「原先資料輸入錯誤，而且尚無任何堂次、薪資或月結紀錄」時，才可以直接修正學生班級而不建立換班事件。跨科目更換不適用本節，必須走原科退班加新科入班，重新計算合報與收退費。

## 6. 老師、科目與薪資規則

- 老師主檔必須有正式顯示名稱與預設薪資規則。
- 每一門課必須明確連到一位老師與一個科目；不能只從課名猜老師。
- 同一位老師可能有舊分頁名、正式姓名或別名，確認後要收斂成同一個老師主檔，歷史顯示名稱可保留作別名。
- 課程要分開保存「實際授課老師」與「薪資歸屬老師」。例如學測生物實際由許昱老師授課、學測地科實際由昱維老師授課，但兩科薪資均歸在周逸老師名下；這是已確認的正確關係，不可把其中一個欄位覆蓋另一個。
- 課程可以覆寫老師預設薪資規則，但必須有原因與稽核紀錄。
- 多數課程目前是固定人頭堂薪，不是按學生實收拆帳：一般每人每堂 `335`，自然科學班常見 `450`。
- 明軒數學為特殊公式：`保底鐘點 + max(當堂人數 - 15, 0) × 300`。
- 國文或其他鐘點課若沒有已確認規則，列為手動覆核，不得套用一般人頭堂薪。
- 每次月結前都要檢查課程的老師／科目對應；不明確或可疑對應不得直接出正式薪資。

## 7. 月結輸出與不可變快照

單一老師薪資條至少包含：

- 月份、老師、薪資規則。
- 各課程名稱、每堂日期、當堂有效人數、單堂薪資與小計。
- 本月入班／退班學生姓名、日期及課程。
- 特殊堂次與待覆核警示。
- 老師小計、製表／覆核／付款確認欄。

全部老師總表要顯示每位老師小計與當月總額。完成覆核後儲存月結快照；日後即使學生名單、堂次或薪資規則改變，也不能自動改寫已送出或已付款的舊快照。若需更正，建立新版本並保留原因。

## 8. 寫入與安全邊界

- 日常新增、加科、退班與退費使用 `bearhigh/accounting/manual/*` 的 scoped multi-location update，確保相關紀錄一起成功或一起失敗。
- 不以整包 `PUT` 覆蓋 `/accounting`，不新增長駐的大節點 listener。
- 不碰 `sciencebear-admin`、`bear-admin/high`、Firebase Rules、Functions、Storage 或 LINE，除非另有明確任務與授權。
- 正式刪除、整批改老師、整批改薪資規則、覆蓋遠端資料、部署與發通知都屬高風險動作，必須先列出影響範圍並取得確認。
- 歷史資料以封存或版本化處理，不硬刪；重複學生只有在已確認永久 ID 與關聯資料後才能合併。

## 9. 其他視窗接手清單

接手 BearHigh 日常帳務前必須：

1. 讀本文件、`docs/numbers_workflow.md`、專案 `README.md` 與 `docs/systems/high-and-bearhigh.md`。
2. 確認工作目標是查核、改程式、改正式資料或部署；不得混為一談。
3. 先看實際子儲存庫的 `git status --short`，保留既有變更。
4. 涉及 Firebase 時，先說明讀寫節點、是單筆／單班／scoped read 還是整包資料，以及費用風險。
5. 老師／科目對應、特殊套餐、特殊堂次與同名學生只要不唯一就停止並請使用者確認。
6. 驗證至少包含公式測試、JavaScript 語法、`git diff --check`；UI 改動另做桌面與手機 smoke test。
